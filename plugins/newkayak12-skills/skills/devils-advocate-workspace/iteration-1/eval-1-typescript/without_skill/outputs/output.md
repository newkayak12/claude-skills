# TypeScript 전환에 대한 반론 3가지

다음 분기에 TypeScript로 전환하기로 결정했다는 소식을 들었습니다. TypeScript의 장점은 분명히 존재하지만, 아래 3가지 반론을 신중히 검토해 보시길 권장합니다.

---

## 반론 1: 단기적 생산성 저하와 전환 비용

TypeScript 전환은 단순히 코드에 타입 주석을 추가하는 작업이 아닙니다. 기존 JavaScript 코드베이스를 TypeScript로 마이그레이션하는 데는 상당한 시간과 노력이 필요합니다.

- **러닝 커브**: 팀원 중 TypeScript에 익숙하지 않은 개발자는 제네릭, 유틸리티 타입, 타입 추론 등을 학습하는 데 시간이 소요됩니다.
- **기존 코드 마이그레이션**: 기존 `.js` 파일을 `.ts`로 전환하면서 발생하는 타입 에러 수천 개를 처리해야 합니다.
- **서드파티 라이브러리 호환성**: `@types/*` 패키지가 없거나 불완전한 라이브러리를 사용할 경우, 직접 타입 정의를 작성해야 합니다.

**핵심 위험**: "다음 분기"라는 짧은 기간 안에 전환을 강행하면, 정작 비즈니스 가치를 만드는 기능 개발 속도가 크게 떨어질 수 있습니다.

---

## 반론 2: TypeScript가 버그를 완전히 막아주지 않는다

TypeScript가 타입 안정성을 제공한다는 것은 사실이지만, 이를 "버그 없는 코드"와 동일시하는 것은 과도한 기대입니다.

- **`any` 타입 남용**: 팀이 TypeScript에 익숙하지 않거나 마이그레이션 압박이 클 경우, `any`를 남발하여 사실상 타입 검사를 무력화시키는 코드가 양산될 수 있습니다.
- **런타임 오류는 여전히 발생**: TypeScript는 컴파일 타임에만 동작합니다. 외부 API 응답, 사용자 입력 등 런타임에 들어오는 데이터의 타입은 TypeScript만으로 보장할 수 없습니다.
- **타입이 곧 테스트가 아님**: 타입 시스템이 있다고 해서 단위 테스트, 통합 테스트의 필요성이 줄어드는 것은 아닙니다. 팀이 "TypeScript 쓰니까 괜찮겠지"라는 안도감에 빠질 위험이 있습니다.

**핵심 위험**: TypeScript 도입에 투자한 비용만큼 실질적인 품질 향상이 이루어지지 않을 수 있으며, 이는 투자 대비 효과(ROI)에 대한 의문으로 이어집니다.

---

## 반론 3: 팀 규모와 프로젝트 성격에 따른 오버엔지니어링

TypeScript는 대규모 팀과 장기적으로 유지보수해야 하는 대형 프로젝트에서 그 가치가 극대화됩니다. 모든 상황에 적합한 것은 아닙니다.

- **소규모 팀 / 빠른 이터레이션**: 팀이 소규모이고 서로 코드를 잘 알고 있다면, 엄격한 타입 시스템보다 코드 리뷰와 테스트가 더 효율적일 수 있습니다.
- **프로토타입 / 실험적 단계**: 제품이 아직 방향성을 잡는 단계라면, TypeScript의 엄격함이 오히려 빠른 실험과 피벗을 방해할 수 있습니다.
- **빌드 파이프라인 복잡도 증가**: TypeScript 컴파일러(`tsc`), `tsconfig.json` 설정, 소스맵 관리 등 빌드 인프라가 복잡해지며, 이는 CI/CD 속도와 디버깅 환경에도 영향을 줍니다.

**핵심 위험**: 현재 팀의 상황과 제품 단계를 고려하지 않고 업계 트렌드만 따라 도입하면, 실질적인 이점 없이 복잡성만 늘어날 수 있습니다.

---

## 결론

TypeScript 전환 자체가 나쁜 결정이라는 것이 아닙니다. 다만, "다음 분기"라는 일정 압박 하에 서두르기보다는 다음을 먼저 검토해 보시기를 권장합니다:

1. 팀 전체의 TypeScript 역량 수준 파악 및 학습 시간 확보
2. 전체 마이그레이션보다 신규 모듈부터 점진적 도입 전략 수립
3. 도입 목표(어떤 문제를 해결하려 하는가)를 명확히 정의하고 성공 지표 설정
